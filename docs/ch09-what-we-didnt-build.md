# Ch9 · 收尾:我们刻意没做的事

> 本章对应 `day8`:没有新能力,只有打磨(npm 封装、`npx agent-from-zero` 一行启动——原想用 mini-agent 当包名,被 npm 的相似名保护拦了:已存在 miniagent;装完的命令名仍是 `mini-agent`)和这份「不做清单」。

## 砍功能也是工程

先讲一个反直觉的工程判断,来自 Cursor 团队的实测:当工具调用可靠性从 99% 提升到 99.9%,他们做的不是加更多防御代码,而是**删掉一层防御代码**——因为防御代码自身也有故障率、也吃上下文、也加延迟,可靠性够高之后,防御代码出问题的概率反超它要防的问题。

「加功能」永远比「不加」容易写进 PPT。但一个 1200 行、每行都看得懂的 agent,比一个 12000 行、作者自己都怕碰的 agent 值钱得多——尤其当它的目的是让你**理解**。所以这最后一章,把那些我们没做的东西摊开:不做的理由,和哪天该做的判断标准。每一条都对应 Claude Code 里一个真实存在的系统——迷你版没做,不代表它们不重要,恰恰因为它们重要到值得你毕业后自己去拆。

## 不做清单

### 1. Hook 生命周期系统

Claude Code 在 26 个时机点允许外部脚本介入:工具调用前、调用后、会话开始、压缩前后……比如「每次 git commit 前自动跑 lint」就是一个 PreToolUse hook。

**为什么不做**:hook 是给「别人扩展你的系统」用的。现在用户就是你自己,改源码比设计扩展点快十倍。
**何时该做**:当第二个人开始用你的 agent,而他的需求你不想合进主干时。到时候记住一条:hook 的 allow 永远不能覆盖用户的 deny(Ch5 的不变量,在扩展系统里同样是地基)。

### 2. 长期记忆

我们的 agent 是金鱼:会话结束,一切归零。Claude Code 有一套六层记忆系统——每轮对话后台提取值得记的事实,凌晨低谷期还有「做梦」任务整理合并记忆文件。

**为什么不做**:记忆系统的收益曲线在「高频使用同一项目」时才起飞,教学场景里它是纯复杂度。而且我们已经有了 80 分的替代品:AGENT.md(Ch7)——把该记住的写成文件,本来就是最稳的记忆。
**何时该做**:当你发现自己每个会话开头都在重复说同样的背景。先从最笨的开始:让 agent 把要点追加进 AGENT.md,就这么简单。

### 3. Skill / 插件系统

Claude Code 的插件不是代码,是 Markdown:一份 SKILL.md 描述触发条件和步骤,模型读了就会执行。门槛低到「会写文档就会写插件」。

**为什么不做**:插件生态需要「生态」,一个人用的工具没有生态。
**何时该做**:当你发现某类任务的提示词反复在对话里出现第三次——把它抽成文件,按关键词注入,就是最小的 skill 系统,50 行能写完。

### 4. 多 Agent 协作(Teams)

我们的子 agent 是「一层分身」:派出去、干完活、交报告、消失。Claude Code 的完整形态是常驻的 agent 团队,靠**文件夹通信**(收件箱是 append-only 的 JSON 文件)协作,任务是带依赖关系的 DAG,空闲 agent 自动认领。

**为什么不做**:单 agent 的能力你还没榨干。多 agent 把所有问题都放大:成本 ×N、失败模式 ×N²、调试难度爆炸。Claude Code 给它定了三条铁律(别偷看、别抢跑、别外包理解),每一条背后都是真实事故。
**何时该做**:当任务真的能并行(比如同时改五个互不相干的模块),且你已经被单 agent 的串行速度卡住。判断标准是吞吐量,不是酷炫。

### 5. LLM 当权限裁判

我们的权限是纯规则(正则 + 名单)。Claude Code 在规则之上还有一层「AI 裁判」:拿不准的命令丢给一个 64 token 的快速模型判断危险性,判完不放心再升级给强模型复审。

**为什么不做**:规则已经兜住了约 95% 的情况(白名单放行 90%,黑名单直拦 5%),裁判只服务最后那点模糊地带——而它自己引入新的失败面(裁判被 prompt 注入骗了怎么办?Claude Code 的对策是给裁判看的内容剥掉所有助手文本,只留工具调用)。**第一道防线必须是确定性代码**,这是 Ch5 说过的,现在你知道第二道在哪了。
**何时该做**:当「不认识的命令一律问人」开始频繁打断你,而那些命令明明无害。

### 6. 可观测性与埋点

Claude Code 埋了 860+ 个遥测事件,从缓存命中率到每种错误的分布,全部进数据管道,其中约 38 个接告警。

**为什么不做**:单用户工具,console.log 就是你的可观测性。我们已经做了它的灵魂:错误分类枚举(Ch4)——「rate_limited 从 0 跳到 5%」五分钟定位,靠的是分类,不是埋点平台。
**何时该做**:有第二个用户的那天。届时记住手册的配比:告警只留 5%,全告警等于没告警。

### 7. 精细的缓存管理

Claude Code 为 prompt 缓存做了大量工程:系统提示词分段、静态段禁 if、缓存中断检测、给不可缓存的函数起名 `DANGEROUS_`。我们只做了一条:**前缀稳定**(宪法零会话变量、历史 append-only)。

**为什么够了**:DeepSeek 的缓存是全自动的,前缀稳定就能拿到命中折扣,剩下的优化空间在我们的量级上是几分钱。
**何时该做**:当月账单让你心疼的时候。先去查缓存命中率,通常一个被忽略的动态字段(日期!)就是答案。

## 这份判断力才是课程的产出

回头看这张清单,每一条的结构都一样:**不是「做不了」,是「现在不值得」**——而「何时该做」的判断标准,全部指向同一个东西:真实的使用信号(第二个用户、第三次重复、被卡住的吞吐量、心疼的账单),而不是技术的诱惑。

这是整门课最想交付的东西。五根骨头(loop / tool / context / sub-agent / eval)是知识,知识会过时——模型再强两代,今天的压缩管线可能整个消失。但「每个机制配三件套:重试上限、日志、最便宜优先」「文字管概率,代码管底线」「deny 永远赢」「没有 eval 永远是 demo」这些判断,会跟着你去下一个系统。


八个源文件,约 1200 行,每行一条注释。它不是玩具——有熔断、有权限、有压缩、有考试;它也不是产品——没有记忆、没有插件、没有团队。它是一个**你完全理解的系统**,而这正是它的全部价值。

去改它吧。加一个你需要的工具,改一条你不同意的规则,跑一遍 eval 看看自己改好了还是改坏了——从这一刻起,你不是在学 agent,你在养一个。
