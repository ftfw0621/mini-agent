# 进阶篇阶段综述(截至 Day 26):一张图看懂这台 agent

> 没有新代码。这一章把进阶篇 16 章拼成一张架构全景——你写了这么多,该退后一步,看看它们合起来是什么。

主线十天,你做出了一个**能用**的 agent:一个循环、五件工具、能压缩上下文、能委托分身、能自我评测。进阶十六天,你把它打磨成了一个**敢用**的 agent。这一章不教新东西,只回答一个问题:这 16 章,到底在主线那个核心外面,套了几层壳?

## 全景图

```
┌─────────────────────────────────────────────────────────────┐
│  你(终端 REPL)                                               │
│   输入 → @file 展开(D26) → slash 命令(D11) → 历史          │

├─────────────────────────────────────────────────────────────┤

│   model → tool → model → 终止(具名状态机)                    │
│   · 重试/退避/熔断(D3)      · 上下文压缩(D5)                │
```

读法:从上往下,是「人 → 编排 → 三根支柱(权限/工具/Hook)→ 配置」。进阶篇做的事,就是把主线那条光秃秃的 `model→tool→model` 循环,从上下两头包起来。

## 四条暗线

16 章不是 16 个孤立功能。把它们重新分组,会浮出四条贯穿始终的设计主张。

### 暗线一:一个闸门,很多入口

这是全篇复用最狠的一条。`checkPermission`(D4)是**唯一**判断「这个动作能不能做」的地方,而后面每一章都在给它加新入口、却从不另起炉灶:

- D15 的 MCP 工具、D24 的远程 MCP 工具,走同一个闸门(默认 ask);
- D16 的 LLM 裁判,只在闸门判 ask 的灰区里、只能 ask→allow,永不碰 deny;
- D20 的计划模式,是闸门外面的一层过滤器,只收紧不放松;
- D26 的 `@file`,引文件前先问闸门 read_file 准不准读,密钥文件照样拒。

**入口可以无限多,裁决只有一处。** 这是整台 agent 安全性的地基:你只需要信任一个函数,而不是十几个散落各处的检查。

### 暗线二:人始终在回路里,而且看得见

agent 的安全感不来自「相信模型不犯错」,来自**在它和真实世界之间一道道架闸**,并且每道闸都让人看得见、插得进手:

- 写文件前,diff 预览(D21)把每一刀摊给你看;
- 改错了,`/undo`(D22)一键退回;
- 收工前,`/diff`(D25)看整趟会话的净改动;
- 拿不准要不要让它动手,计划模式(D20)让它先交计划等你批。

D21 看单刀、D22 撤单刀、D25 看整程、D20 先看计划——人对 agent 的掌控,从动手前延伸到了动手后。

### 暗线三:失败是状态,不是异常

模型是概率系统,网络会抖,工具会崩。进阶篇把每一种「出错」都变成了**一个要处理的具名状态**,而不是一个让程序崩掉的异常:

- D3 的调用层:重试、退避、熔断,每种死法一个 `TerminateReason`;
- D19 的工具层:参数不合规,翻译成「Fix and call again」喂回模型,让它自助修复;
- D17 的记忆、D21 的预览、D26 的引用:全都「永不抛异常,fail 到一个安全值」。

错误是给模型(或给人)的操作指南,不是终点。这条线把可靠性焊在了概率系统的两端:输入有 D19 约束,调用有 D3 兜底。

### 暗线四:把易错的纯逻辑拎出来单测

你可能没注意,但进阶篇有个一以贯之的代码习惯:凡是「难单测的流程里藏着一段易错的纯判断」,就把那段拎成纯函数:

- D16 `interpretJudgeOutput`、D19 `validateArgs`、D23 `subAgentModelFor`、D24 `parseSseData`……

它们都不碰网络、不碰真实 API,给输入、要输出,于是能被几十个边界用例钉死。而真正难测的部分(发 API 请求的 `runLoop`),D23 用一个**假 client** 记录每次调用的 model、D24 用一个**临时 HTTP server** 跑通远程路径——不联外网,却把真实路径端到端验证。`npm test` 从主线的几十例长到 244 例,靠的就是这个习惯。

## 哪些是「壳」,哪些是「核」

退一步分层,会更清楚进阶篇的取舍:

- **核(主线 Ch1-9)**:循环、工具、上下文、子 agent、eval。没有它们,agent 不存在。
- **可靠性壳**:D3 重试、D13 异步、D19 校验。让它在真实世界的抖动里不倒。
- **安全壳**:D4 权限、D16 裁判、D20 计划、D21/22/25 改动可视可撤、D26 引用守密。让它敢交给它动你的文件。
- **可运维壳**:D11 配置、D12 持久化+遥测、D18 成本。让你看得见它在干什么、花了多少。
- **可扩展壳**:D14 Hook、D15/24 MCP、D23 分级。让它接得上你的工具、你的策略、你的模型组合。

主线是「能跑」,四层壳是「能放心长期用」。工业级和玩具的差距,基本都在这四层壳里——而每一层,都没有动核心那条循环。**好的扩展是加壳,不是改核。**

## 你还能往上加什么

这台 agent 远没到头。沿着已经铺好的地基,下一步很自然:

- **`/resume` 选会话**:D12 已经把会话存成了快照,列出来让人挑一个续,只差一个 UI;
- **流式渲染 markdown**:D6 已经在流式输出,把 token 流接一个增量 markdown 渲染器即可;
- **Advisor 自动审查**:D23 已经能让强模型当子 agent,D16 已经有裁判范式,合起来就是「强模型自动复核弱模型的产出」;
- **per-model 计价**:D18 的成本表目前按主模型单价混算,D23 引入了多模型——把价格表做成 per-model 就精确了。

每一条都不是从零开始,而是把已有的两三块拼起来。这正是分层架构的回报:**地基铺对了,新功能是组合,不是重写。**

## 收尾

主线教你**搭起一个 agent**,进阶篇教你**把它变得敢用**。回头看,进阶篇没有一章在改那条 `model→tool→model` 的核心循环——它们全是在循环外面,一层层加闸门、加可视、加兜底、加扩展点。

如果只让你带走一句话:**agent 工程的难点不在让模型动起来,在于在模型和真实世界之间,架起足够多、且人看得见的确定性的闸。** 主线让它动,进阶篇架那些闸。

这就是这台从零搓出来的 mini-agent,26 天、12 个源文件、244 个测试,该有的工业级骨架,都在了。
